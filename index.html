PART 1

什麼是 Claude Skills？

想像你剛請了一位很聰明的新助理。他很厲害，但第一天完全不了解你——不知道你的工作流程、你喜歡的格式、你的觀眾喜歡什麼、你的品牌怎麼說話。所以每一天，你都要花 20 分鐘重新解釋一遍，才能真正開始做事。

Claude Skills 就是為了解決這個問題而生。

一個 Skill 其實就是一組指令，包裝成一個簡單的資料夾。你上傳之後，Claude 會在相關的時候自動讀取它——不用提醒、不用重新解釋、不用每次從零開始。

可以把 Skills 想成「Claude 的模組化升級包」。你把自己的做事方法寫下來一次，Claude 之後每次對話都會延續使用。

真正厲害的地方在於背後的機制：Skills 使用「漸進式揭露（progressive disclosure）」。Claude 不會把你完整的指令內容塞進每一次對話，而是先讀一段簡短描述——剛好足夠知道這個 Skill 存在。如果你的需求對得上，Claude 才會載入完整指令；對不上，就完全忽略。這代表你可以安裝 50 幾個 Skills，Claude 依然跑得很快，不會佔用過多的上下文空間。

Skills 可以跨 Claude.ai、Claude Code、API 通用——寫一次，到處都能用。

PART 2

6 大效益，Skills 幫你解鎖的事

每次跟 Claude 開新對話，預設都是從零開始。它不記得你怎麼工作、你做過什麼、你在意什麼。Skills 從根本上改變了這件事，實際能解鎖的效益包括：

1

不用再重複解釋

不論是品牌語氣、研究流程還是文件格式，你在 Skill 裡解釋一次就好。之後每次對話，Claude 都已經知道。

2

每次輸出都一致

同一組指令，每個 session、每位團隊成員執行結果都一樣。在一個本質上非決定性的系統裡，Skills 是最接近「品質保證」的東西。

3

工作流程更快，效益會複利

有重度使用者回報，每週兩小時的規劃會議縮短到 10 分鐘；也有人每天省下 30–45 分鐘的重複性工作。Skill 用得越多，回報的時間就越多。

4

Claude 自動選對的 Skill

你不需要說「現在用我的品牌語氣 Skill」。Claude 讀懂你的需求，自動辨識出符合的 Skill，並在背後默默套用。

5

裝再多個都不會變慢

因為漸進式揭露的機制，就算裝了 50 個 Skills，對效能幾乎沒有影響。Claude 只會載入真正需要的部分。

6

一個重要觀念：Skills ≠ Projects

Projects 存放的是背景知識——文件、脈絡、歷史紀錄；Skills 存放的是做事流程——怎麼做。Projects 說「這是你需要知道的」，Skills 說「這是該怎麼做的」。兩者搭配使用效果最好。

PART 3

如何寫一個 Claude Skill

一個 Skill 就是一個資料夾，裡面只需要一個必要檔案：SKILL.md。完全不用寫程式。這個檔案分成兩個部分。

Step 1．YAML Frontmatter（標籤區）

放在檔案最上方，夾在兩組 --- 之間。內容雖短，卻是整個 Skill 最關鍵的部分。

EN · 直接照格式使用 
---
name: brand-voice
description: Applies a casual, punchy, jargon-free voice to any written output. ALWAYS invoke this skill when writing blog posts, emails, social copy, newsletters, or any customer-facing content.
---

這是 Claude 用來判斷「要不要載入你的 Skill」的依據。Claude 會掃過每一個已安裝 Skill 的 description，決定哪些跟目前任務有關。如果你的描述很模糊，Claude 會悄悄忽略這個 Skill——就算它其實完全符合需求。

研究顯示：被動、模糊的描述，觸發率可能低到只有 37%；具體、指令式的描述，觸發率可以接近 100%。

描述（description）要遵守的規則：

  • 同時說明「做什麼」和「什麼時候用」——兩者缺一不可
  • 加入具體的觸發語句（例如「when writing newsletters」「when user asks for a proposal」）
  • 用第三人稱寫（「Applies brand voice」而不是「I help with writing」）
  • 語氣可以強硬一點——像「ALWAYS invoke this skill when...」這種寫法，比委婉建議更容易穩定觸發
  • 如果是給 Claude.ai（消費版）用，描述控制在 200 字元以內；若是用 API，可以到 1,024 字元

命名（name）要遵守的規則：

  • 只用小寫字母，字詞之間用連字號：brand-voice ✅
  • 不能有空格、不能有大寫、不能用底線：Brand Voice ❌

Step 2．Markdown Body（說明書本體）

Frontmatter 之後的所有內容，用一般的 Markdown 撰寫。這裡才是你真正告訴 Claude「該怎麼做」的地方。建議結構：

  • Overview 總覽——一段話說明這個 Skill 做什麼、解決什麼問題
  • Step-by-step instructions 逐步指示——帶著 Claude 一步步做，把 Claude 不會自己知道的規則寫清楚
  • Examples 範例——給 Claude 看 before/after 或 input/output 的對照，這是最有效的引導方式之一
  • Quality checklist 品質檢查表——完成前 Claude 可以自我檢查的幾個重點
  • Limitations 限制——這個 Skill「不適用」的場合，避免 Claude 用錯地方
  • Keep it lean 保持精簡——主要的 SKILL.md 盡量控制在 500 行以內。如果有詳細參考資料（完整品牌手冊、風格指南、範本），移到 references/ 子資料夾，Claude 只有真正需要時才會載入。
重點提醒

3 個最常見的錯誤

① 描述太模糊 「Helps with content」這種寫法永遠不會穩定觸發。「Applies a punchy, jargon-free brand voice when writing newsletters, social posts, and landing pages」才會。如果你的 Skill 該觸發卻沒觸發，問題幾乎都出在 description 上。
② 一個 Skill 塞進所有功能 這樣的 Skill 觸發不穩定，還會讓 Claude 混亂。原則是：一個 Skill，一個任務。需要多功能時，就建立多個 Skill，讓它們彼此搭配使用。
③ 過度解釋 Claude 早就知道的事 不用定義「什麼是部落格文章」。只需要告訴 Claude 你自己的規則——那些跟你的做事方式有關、Claude 不會自己知道的部分。
新手推薦

最快建立第一個 Skill 的方法

  1. 正常跟 Claude 對話，實際做一次你想要的任務，用你喜歡的方式做
  2. 持續微調，直到輸出結果完全符合你要的樣子
  3. 接著跟 Claude 說：「把這次對話學到的東西，整理成一個 Skill 檔案」
  4. Claude 會直接產出格式化好的 SKILL.md
  5. 下載檔案、放進資料夾、壓縮資料夾，再到 Settings → Capabilities → Skills 上傳
測試技巧

用一個「真實、有點隨性」的提問來測試，不要用完美措辭的句子——就用你平常忙碌時實際會打的那種句子。如果沒有觸發，就把 description 寫得更具體、加入更多觸發語句，語氣也更直接一點。

操作步驟

快速安裝：3 步驟裝好你的第一個 Skill

  1. 開啟 Skills 功能 —— 到 Settings → Capabilities，打開「Code execution and file creation」。Skills 需要這個功能才能運作。
  2. 建立你的 Skill 資料夾 —— 在電腦上新增一個資料夾，用小寫＋連字號命名（例如 brand-voice）。資料夾裡建立一個檔名完全是 SKILL.md 的檔案（區分大小寫）。貼上你選好的 Skill 範本，把裡面的括號段落改成你自己的內容。
  3. 上傳到 Claude —— 把資料夾壓縮成 zip，到 Settings → Capabilities → Skills → 點「+」→ 上傳 zip 檔 → 開啟這個 Skill 的開關。

完成後，下一次對話就會自動用上這個 Skill。

PART 3 延伸

5 個能自動幫你工作的 Skills

以下每個 Skill 都附上兩樣東西：可以直接貼給 Skill Creator 的英文提示詞，以及完成後的 SKILL.md 內容，方便你直接參考或修改。指令與程式碼部分保留英文原文，貼上就能用。

SKILL 1

Brand Voice 品牌語氣

把你具體的語氣、句型、用字規則，套用到任何文字輸出上——讓 Claude 寫出來的東西聽起來像你，而不是通用 AI 腔調。教一次，之後每次寫作都自動套用。

需要準備的輸入：
  • 用幾個詞形容你的語氣（例如：口語、直接、溫暖）
  • 句子長度與節奏偏好
  • 你常用的詞彙與說法
  • 你絕對不會用的詞彙與說法
  • 1–2 篇你的文字範例，讓 Claude 抓到你的節奏
怎麼使用：

給 Claude 一個主題、草稿或內容需求，它就會自動套用你的語氣。適用部落格、電子報、社群文案、Email、Landing Page 等所有面對讀者的內容。語氣規則寫得越具體，輸出越穩定；只要發現輸出走鐘，隨時回來調整規則就好。

展開：貼給 Skill Creator 的英文提示詞 
Use the skill creator to build me a skill that applies my brand voice to any written content. I should be able to give it a topic, a draft, or a brief, and it should write in my specific tone and style every time. It should follow a set of voice rules I define: my tone in a few words, the sentence length and rhythm I use, the vocabulary I reach for, and a list of words and phrases I never use. It should always do certain things — like lead with the benefit and use active voice — and never do others, like write long dense paragraphs or use corporate buzzwords. Before finishing any piece of content it should run a quick self-check: does this sound like a real person, not a press release? Keep it consistent every time, regardless of the topic.
展開：完整 SKILL.md 內容（英文） 
---
name: brand-voice
description: >
Applies [YOUR BRAND NAME]'s specific tone, sentence style, and
voice rules to any written output. ALWAYS invoke this skill when
writing blog posts, social media posts, emails, newsletters,
landing pages, or any customer-facing copy. Also trigger when the
user says "write this in my voice," "make this sound like me," or
"apply our brand voice to this."
---

# Brand Voice

## Why this exists
Generic AI output sounds like everyone else. This skill makes sure
Claude writes in your voice — not a default one. The rules here are
the things Claude would not know without you: your rhythm, your
vocabulary, your hard limits. Give it a topic or a draft and it
writes as you, not as an AI approximation of a writer.

## Voice rules

### Tone
[e.g., Conversational and confident. Write like a smart friend who
has done the research, not a corporate spokesperson.]

### Sentence style
[e.g., Short sentences. One idea per sentence. Vary the rhythm —
some punchy two-word lines, some longer ones for context. Never more
than three sentences in a paragraph.]

### Vocabulary
[e.g., Plain English. Use the words your audience actually says.
If you would not say it out loud in a meeting, do not write it.]

### Always do
- [e.g., Lead with the benefit, not the feature]
- [e.g., Use active voice — subject does the action]
- [e.g., End every piece with a clear next step]
- [e.g., Use specific numbers and examples over vague claims]

### Never do
- [e.g., Use buzzwords: "synergy," "leverage," "holistic," "ideate"]
- [e.g., Open with "In today's fast-paced world" or any variation]
- [e.g., Write paragraphs longer than 3 sentences]
- [e.g., Use the em dash heavily — it signals AI writing]
- [e.g., Use passive voice]

## Writing samples
[Paste 1–2 short pieces you wrote and consider on-brand. Claude
will match the rhythm, not just the rules.]

## Self-check before finishing
- Does this sound like a real person talking, not a press release?
- Are all paragraphs under 3 sentences?
- Is every sentence earning its place, or is something padding?
- Would the target reader recognize this as written for them?

## What this skill is not for
Technical documentation, legal copy, or formal reports where a
different register is required. For those, disable this skill.
SKILL 2

Audience Profile 受眾輪廓

讓 Claude 對「你在寫給誰看」有深入且具體的理解——他們的用詞、他們的痛點、他們的動機、什麼會讓他們真的採取行動。跟 Brand Voice 搭配使用，每篇內容都會因為寫給真實的人，而更有說服力。

需要準備的輸入：
  • 他們是誰：角色、階段、對 AI 的熟悉程度、日常情境
  • 他們在意什麼：目標、優先順序、想要的結果
  • 他們的用詞：會用的字 vs. 不會用的字
  • 什麼讓他們感到困擾（對工具、內容、廠商）
  • 什麼會驅動他們行動：急迫感、情緒觸發點
怎麼使用：

跟 Brand Voice 一起安裝，兩者會自動搭配運作——Brand Voice 控制「怎麼說」，Audience Profile 控制「說給誰聽」。受眾改變或你進入新的市場區隔時，隨時更新內容。痛點跟用詞寫得越具體，輸出就越不會像是「寫給任何人都可以」的通用內容。

展開：貼給 Skill Creator 的英文提示詞 
Use the skill creator to build me a skill that gives Claude a detailed profile of my target audience. I should be able to define who they are in terms of their role and situation, what they care about most, the exact vocabulary they use and the words they avoid, what frustrates them, and what makes them click or buy. The skill should use this profile any time Claude is writing content, copy, or messaging for me — so it always writes to the right person, not a generic reader. Include a section for what motivates them to take action, since that is the most important part to get right.
展開：完整 SKILL.md 內容（英文） 
---
name: audience-profile
description: >
Loads a detailed profile of [YOUR AUDIENCE NAME] so all content,
copy, and messaging speaks directly to the right person. ALWAYS
invoke this skill when writing content, emails, social posts, or
any messaging intended for [YOUR AUDIENCE, e.g., AI enthusiasts,
early-stage founders, B2B marketers]. Also trigger when the user
says "write for my audience," "who is this for," or "make this

# Audience Profile

## Why this exists
Writing for "everyone" means writing for no one. This skill gives
Claude a specific person to write to — their language, their
frustrations, their motivations, and what makes them act. Every
piece of content produced while this skill is active should feel
like it was written for one reader who will think: "this is exactly
my situation."

## Who they are
- **Role:** [e.g., Founders, marketers, and operators at companies
with 1–50 employees]
- **Stage:** [e.g., Post-product, pre-scale. Wearing multiple hats
and trying to do more with less.]
- **AI familiarity:** [e.g., Curious and early-adopting, but not
deeply technical. They want tools, not theory.]
- **Daily context:** [e.g., Busy. Skimming. Making fast decisions.
Will not read a wall of text.]

## What they care about
- [e.g., Getting more done without hiring more people]
- [e.g., Looking ahead of the curve to peers and clients]
- [e.g., Practical tools they can use today, not in six months]
- [e.g., Honest assessments — they distrust hype]

## Their vocabulary
**Words and phrases they use:**
[e.g., "workflow," "stack," "ship," "automate," "leverage AI,"
"build in public," "systems thinking"]

**Words and phrases they don't use:**
[e.g., "synergize," "ideate," "circle back," "deep dive,"
"thought leader," "disruptive"]

## What frustrates them
- [e.g., Tools that require technical setup to get basic value]
- [e.g., Guides that explain what to do but not how]
- [e.g., AI content that sounds like AI]
- [e.g., Overpromising without showing proof]

## What motivates them to act
- [e.g., Feeling like they are ahead of their competitors]
- [e.g., A fast, tangible win they can implement the same day]
- [e.g., Seeing proof that something works before they invest time]
- [e.g., Someone being honest about limitations, not just the upside]

## Writing guidance
Always write as if addressing one smart, busy person. Lead with
value. Use their vocabulary, not yours. Respect their time — if a
sentence does not earn its place, cut it. Avoid anything that could
have been written by anyone; if it is generic, it will be ignored.

## What this skill is not for
Technical documentation or internal communications where audience
context is not relevant.
SKILL 3

Hyper-Personalized Content Engine 內容引擎

把一段影片、Podcast 或語音備忘錄的逐字稿，挖出裡面真正有價值的想法，再轉換成一批可以直接發佈的 LinkedIn 貼文、Twitter/X 串文，以及一篇 SEO 文章——而且全部都用你的語氣。不是摘要，也不是 AI 廢文，是讀起來真的像人寫的一週份內容。

需要準備的輸入：
  • 原始逐字稿（直接貼進對話）
  • 一份填好的 references/brand_bible.md 檔案（見下方範本——只要填一次，之後每次都會套用）
怎麼使用：

先填好一次 brand bible——這是讓每次輸出都聽起來像你、而不是機器的關鍵投資。之後貼上任何逐字稿，Skill 就會挖出真正有趣的重點、強烈的觀點、故事、具體的數字，廢話會被自動捨棄。每種格式都針對該平台寫作：LinkedIn 貼文留白方便掃讀、串文每則獨立成立、文章有清楚標題方便搜尋。錄完影片、上完 Podcast、講完一場演講之後直接跑一次，就有一週份的內容。

# Brand Bible

[Your role, your audience, the topics you own.]

[e.g., direct, warm, practical]

## Phrases and words I use
[Signature phrases, the way you open and close, words you reach for.]

## Words and phrases I never use

[Paste 2–3 short pieces you wrote, so the rhythm can be matched.]

# Content Engine

## Why this exists
Most repurposed content reads like a machine flattened a transcript.
The ideas are buried, the voice is gone, and it sounds like everyone
else. This skill does the opposite: it mines the transcript for the
real ideas, then writes them in the user's actual voice by following
a brand bible the user fills in once. The output should be
indistinguishable from something the user wrote on a good day.

## Workflow

### 1. Load the brand bible
Read references/brand_bible.md first. It defines tone, signature
phrases, banned words, point of view, and writing samples. If it is
missing, ask the user to fill in a short version before writing —
voice is the whole point and generic output without it defeats
the purpose.

### 2. Mine the transcript for ideas
Read the transcript and pull out the genuinely interesting points:
the strong opinions, the stories, the specific numbers and examples.
Discard filler, throat-clearing, and anything that could have been
said by anyone. A transcript usually holds three to six real ideas
worth a post each.

### 3. Write in voice, per format
For each format, match the brand bible rhythm exactly. Lead every
piece with a real hook, not a windup. Keep sentences short and
natural. Avoid hype words and the heavy em dash cadence that signals
AI. Never open with a tired stock phrase. If a line could have been
written by anyone, rewrite it in the user's voice.

### 4. Format for the platform
- **LinkedIn posts:** Short lines and white space for easy reading.
One idea per post. Hook in the first line — it must work before
the "see more" cut.
- **Threads:** Numbered. Each post stands alone. The first tweet is
the hook; the last is the takeaway or CTA.
- **SEO article:** Clear headline and subheadings. Lead with the
answer, not a preamble. Written for search and for humans.

## Output structure

ALWAYS use this structure:

---
## LinkedIn posts
[3–5 posts, each ready to paste. Separated by a blank line.]

## Threads
[2–3 threads, each numbered from 1/.]

## SEO article
[One article with a headline and subheadings.]
---

# Meeting Notes Synthesizer

## Why this exists
A transcript is long, repetitive, and useless as a record. People leave a meeting unsure who owns what. This skill cuts the transcript down to what matters — and most importantly, separates decisions from discussion and pins each action to an owner, so the recap is something the team can act on rather than reread.

## Workflow

### 1. Read for signal
Skim the transcript and separate the substance from the small talk and the thinking out loud. Note where the group actually decided something versus merely floated an idea. These are different things and must be recorded differently.

### 2. Extract decisions and actions
Pull out every decision as a flat, plain statement. Pull out every action item with its owner and due date. If an owner or date was never stated, mark it "unassigned" or "no date set" rather than
inventing one. A wrong owner is worse than a blank — it creates false accountability and nothing gets done.

### 3. Capture open questions and risks
List every question the meeting did not resolve. List every risk or concern someone raised, even if the group did not act on it. These are the things most likely to be forgotten and most likely to matter.

### 4. Write the recap
Keep it short and skimmable. Ready to paste into an email or doc without editing.

## Report structure

ALWAYS use this template:

---
# Meeting Recap: [Topic] — [date]

## Summary
[Two to four sentences on what the meeting was about and what came of it.]

## Decisions
- [Decision stated plainly as a conclusion, not a discussion point.]

## Action items
- [ ] [Task] — Owner: [name or unassigned] — Due: [date or none set]

## Open questions
- [Question still unresolved, and what it is waiting on.]

## Risks and concerns
- [Concern raised, and by whom if known.]
---

Never guess an owner or a date. A blank flagged clearly is more useful than a confident error.

## What this skill is not for
Non-meeting text such as articles or reports. Creating formal minutes for legal or governance purposes. Building detailed project plans from a single meeting.

# YouTube Launch Kit

## Why this exists
Shooting a video is the easy part. The title, description, and chapters decide whether anyone watches it, and assembling them by hand is slow and easy to get wrong. This skill takes the transcript and returns the whole publishing package at once — tuned for the click and for search — so a finished video can go live in minutes.

## Workflow

### 1. Find the hook and the value
Read the transcript and identify the single most compelling promise of the video and the concrete value it delivers. Everything else — titles, description, chapters — follows from this. Get this wrong
and the rest will be off.

### 2. Write the titles
Produce several short, high-curiosity title options. Keep them punchy and specific. Curiosity drives clicks; misleading titles drive away the right audience and tank watch time, which hurts the
channel. Aim for titles a human creator would actually publish.

### 3. Build the description and chapters
Write a description that leads with the value in plain language at an easy reading level, then supports it with enough detail for search to work. Build chapters from where the topics actually change
in the transcript, with real timestamps — not invented ones. If the transcript has no timestamps, note the approximate position in the video based on content flow.

### 4. Round out the kit
Add a list of relevant tags, two to three thumbnail text ideas of three to four words each, and a pinned comment that invites a reply or points to the next logical step for the viewer.

## Output structure

ALWAYS use this template:

---
## Title options
1. [Short, punchy, high curiosity — no clickbait]
2.
3.
4.
5.

## Description
[Value-first opening. Plain language. Search-friendly detail below.]

## Chapters
0:00 Introduction
[mm:ss] [Section title]

## Tags
[Comma-separated, relevant to topic and channel niche]

## Thumbnail text ideas
- [Three to four words]
- [Three to four words]
- [Three to four words]

## Pinned comment
[A line that drives engagement, invites a reply, or points to a next step.]
---

Base chapters on the transcript timing. Keep titles honest. Write the description plainly — it should be readable, not stuffed.

## What this skill is not for
Blog content, social posts, or any content not connected to a video. Podcast show notes should use a separate skill built for that format.

幾個小提醒

  • 從一個開始：挑一個跟你最常重複做的任務最相關的 Skill，先做到穩定好用，再考慮增加其他 Skill。
